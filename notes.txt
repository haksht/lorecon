### ✅ Critical Parsing Bugs Fixed (5 major bugs)

**Bug #1: PacketID Wrong Offset**
- **Was**: Reading bytes 10-13 (wrong!)
- **Now**: Reading bytes 8-11 (correct!)
- **Meshtastic PacketHeader**: To(4) + From(4) + **ID(4)** + Flags(1)...
- **Impact**: All decryption now works

**Bug #2: NodeID Wrong Endianness**
- **Was**: Big-endian conversion (0x44d7a39e)
- **Now**: Little-endian (0x9ea3d744) ← Matches device display !9ea3d744
- **Fix**: `(b[4]) | (b[5]<<8) | (b[6]<<16) | (b[7]<<24)`
- **Impact**: Nonce construction now correct

**Bug #3: TEXT_MESSAGE_APP Extraction Pattern**
- **Was**: Looking for nested 0x0A tag (failed)
- **Now**: Pattern 1 extracts raw payload after length varint
- **Format**: `08 01 12 <len> <text_bytes>` (not nested protobuf!)
- **Impact**: Text messages now display

**Bug #4: False Positive Plaintext Detection**
- **Was**: Wire type check only (25% false positive rate)
- **Now**: Requires 0x08 AND valid portnum (0x01/03/04/07/08/09)
- **Impact**: No more garbage extraction from encrypted data

**Bug #5: Telemetry Portnum Confusion**
- **Was**: Checking portnum 0x01 for telemetry
- **Now**: Checking portnum 0x08 (TELEMETRY_APP)
- **Added**: Battery (0x08 tag), voltage (0x15), channel util (0x1d), air util (0x25)
- **Impact**: Ready to parse telemetry when packets arrive

## 📝 CRITICAL TECHNICAL NOTES

### Meshtastic Packet Structure
```
Bytes 0-3:   To (destination node, little-endian)
Bytes 4-7:   From (source node, little-endian) ← Used for NodeID
Bytes 8-11:  ID (packet ID, little-endian) ← Used for nonce
Byte 12:     Flags
Byte 13:     Channel index
Byte 14-15:  Routing fields
Byte 16+:    Encrypted payload
```

### AES-CTR Nonce Construction
```c
Bytes 0-3:   PacketID (little-endian from header bytes 8-11)
Bytes 4-7:   Padding (0x00)
Bytes 8-11:  NodeID (raw bytes from header bytes 4-7, NOT converted!)
Bytes 12-15: Padding (0x00)
```

### Channel PSK (Key #2)
```
Base64: "1PG7OiApB1nwvP+rz05pAQ=="
Hex:    D4 F1 BB 3A 20 29 07 59 F0 BC FF AB CF 4E 69 01
```

### TEXT_MESSAGE_APP Format
```
08 01          # Field 1 (portnum), varint value 0x01
12 <len>       # Field 2 (payload), length varint
<text_bytes>   # Raw text (NOT nested with 0x0A tag!)
```

## 🎓 KEY LEARNINGS FROM THIS SESSION

1. **Byte Order Matters**: Always verify with actual device displays. Meshtastic uses little-endian throughout (ESP32 native).

2. **Protobuf Validation Tricky**: Wire type alone insufficient for encrypted/plaintext detection. Need semantic checks (field + portnum).

3. **TEXT_MESSAGE_APP is Different**: Uses raw payload format, not nested protobuf with 0x0A tag like other types.

4. **Test with Real Data**: Endianness bugs caught by comparing serial output NodeID (0x44d7a39e) with device display (!9ea3d744).

5. **Watchdog Critical**: Any loop that can run >5 seconds MUST feed watchdog. User input waits are particularly dangerous.

6. **Radio Transmit Blocks**: `radio.transmit()` can take several seconds with high spreading factors. Feed watchdog before/after.

## 💡 DEBUGGING TIPS FOR NEXT SESSION

### If Text Extraction Fails
1. Check decrypted bytes for `08 01 12` pattern (TEXT_MESSAGE_APP)
2. Verify Pattern 1 extraction (raw payload after length varint)
3. Check for 0x0A nested tag (older Pattern 2)

### If Decryption Fails
1. Verify PacketID from bytes 8-11 (little-endian)
2. Check NodeID from bytes 4-7 (little-endian)
3. Print nonce and key bytes for manual verification
4. Confirm payload starts at byte 16

### If Telemetry Not Working
1. Verify portnum is 0x08 (not 0x01!)
2. Look for field tags: 0x08 (battery), 0x15 (voltage), 0x1d (channel), 0x25 (air util)
3. Check validation ranges (battery ≤101%, voltage 0-10V)

### If System Reboots During Replay
1. Check watchdog feeding in all while loops
2. Verify `esp_task_wdt_reset()` called every <5 seconds
3. Look for long-running blocking operations
