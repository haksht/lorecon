# Session Handoff - December 17, 2025

## Problem Statement
Device was spontaneously exiting frequency targeting mode during unattended operation. User wants to run a week-long test monitoring a specific frequency without the mode being changed.

## Root Cause Investigation
Multiple potential causes were identified:
1. **Watchdog timeout reboots** - Device rebooting and losing mode state
2. **Serial noise** - Garbage bytes on serial line being interpreted as commands
3. **Unknown triggers** - Mode changes happening without user interaction

## Fixes Implemented

### 1. Mode Persistence via NVS (firmware/src/lora_recon_tool.cpp)
- Added `#include <Preferences.h>`
- Mode is saved to NVS when entering targeted capture or frequency targeting
- Mode is restored on boot if valid
- Cleared when explicitly returning to recon mode

**Key code locations:**
- `initialize()` ~line 88: Restores persisted mode on boot
- `startTargetedCapture()` ~line 354: Saves mode to NVS
- `startFrequencyTargeting()` ~line 434: Saves mode to NVS

### 2. Serial Noise Filtering (firmware/src/command_handler.cpp)
- `handleCommand()` now rejects non-printable characters (outside 0x20-0x7E range)
- Logs rejected bytes as `[SERIAL-NOISE]` for debugging
- Valid commands logged as `[SERIAL-CMD]`

### 3. Restart Reason Logging (firmware/src/main.cpp)
- Added `#include <esp_system.h>`
- Logs `esp_reset_reason()` on boot with human-readable descriptions
- Flags abnormal restarts (PANIC, TASK_WDT, INT_WDT, BROWNOUT) with warnings

### 4. Mode Clearing on Resume (multiple files)
- `command_handler.cpp`: `cmdResumeRecon()` clears NVS
- `api_controller.cpp`: `startScan()` clears NVS
- `recon_service.cpp`: `stopCapture()` clears NVS

## Files Modified
- `firmware/src/main.cpp` - Restart reason logging

## What Was NOT Changed

## NVS Namespace Details
- Namespace: `"mode"`
- Keys:
  - `"mode"` (uint8_t): MODE_RECONNAISSANCE or MODE_TARGETED_CAPTURE
  - `"targetCfg"` (uint8_t): Config index for frequency targeting
  - `"byDevice"` (bool): Whether targeting was triggered by device selection

1. Set frequency targeting via web UI
2. Reboot device (power cycle or reset button)
3. Device should restore targeting mode on boot
4. Serial output shows: `🔄 Restoring persisted targeting mode (config X)`

- Watchdog timeout (30s) is enabled - if main loop blocks, device will reboot
- Mode now persists across reboots, so even WDT reboots won't lose the mode

## December 18, 2025 Updates (Repository Pattern Refactoring)

### Repository Layer Created

New directory `firmware/src/repositories/` with focused storage classes:

#### DeviceRepository (`device_repository.h/cpp` - 378 lines)
- Manages `TargetableDevice` storage and retrieval
- Implements Welford's algorithm for running RSSI statistics
- Supports custom device identification via callback
- Methods: `addOrUpdate()`, `findByNodeId()`, `getByIndex()`, `clear()`, `count()`

#### PacketStore (`packet_store.h/cpp` - 159 lines)
- Manages captured packets for replay functionality
- Fixed-size circular buffer (10 slots per `Config::Replay::MAX_SLOTS`)
- Methods: `capturePacket()`, `getPacket()`, `clear()`, `count()`, `isFull()`

#### NodeTracker (`node_tracker.h/cpp` - 166 lines)
- Tracks "hot" nodes for behavioral analysis
- Running average RSSI with overflow protection (caps at UINT16_MAX)
- Methods: `updateNode()`, `findNode()`, `getByIndex()`, `clear()`, `count()`

All repositories share common patterns:
- Iterator support (`begin()`/`end()`) for range-based for loops
- Capacity checking (`isEmpty()`, `isFull()`, `capacity()`)
- Logging via `LOG_INFO`/`LOG_WARN` macros

### ReconState Integration

`ReconState` now delegates to repositories internally while maintaining backward-compatible public API:

```cpp
class ReconState {
private:
    PacketStore packetStore_;
    NodeTracker nodeTracker_;
    DeviceRepository deviceRepo_;
public:
    // Public API unchanged - delegates to repositories
    void addTargetableDevice(...) { deviceRepo_.addOrUpdate(...); }
    bool capturePacketForReplay(...) { return packetStore_.capturePacket(...); }
    void updateNode(...) { nodeTracker_.updateNode(...); }
};
```

Legacy public arrays (`replaySlots`, `trackedNodes`, `targetableDevices`) kept in sync for consumers that access them directly.
